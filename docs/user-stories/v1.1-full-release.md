# V1.1 (Full Release) — User Stories

Completes the spec vision on top of the MVP: Equinox class ingestion and slotting, injury identification assist (post legal review), proactive temporal check-ins, multi-wearable support, advanced healthspan programming, periodization and plateau-driven re-planning, richer privacy controls, and platform observability.

Story IDs continue the numbering from `v1-mvp.md` within each epic. See `README.md` for conventions and personas.

---

## E1 — Authentication, Accounts & Privacy (enhancements)

### E1-S7 — Social / federated login `[P1]`
As a **Trainee**, I want to sign in with Google, so that I can onboard faster without a new password.

- Google sign-in creates or links an account and restores Coach Memory.
- Federated identities can be linked/unlinked from an existing email account.

### E1-S8 — Export my data `[P1]`
As a **Trainee**, I want to export my profile, Coach Memory, and history, so that I own a portable copy.

- Export produces a machine-readable file (e.g. JSON) covering profile, memory, injuries, diet, and workout logs.
- Export is generated on request and delivered securely.

### E1-S9 — Granular health-data consent `[P1]`
As a **Trainee**, I want granular control over which signals are ingested (sleep, RHR, HRV), so that I share only what I'm comfortable with.

- I can toggle individual signal categories on/off.
- Disabling a signal removes it from readiness computation and explains the impact.

### E1-S10 — Manage active sessions and devices `[P2]`
As a **Trainee**, I want to see and revoke active sessions on my devices, so that I can secure my account if a device is lost.

- A device/session list shows last activity; revoking a session forces re-login there.

## E2 — Adaptive Onboarding & User Model (enhancements)

### E2-S9 — Deep adaptive expansion `[P1]`
As a new **Trainee**, I want onboarding to branch into deeper questions only when my answers justify it, so that complex profiles are captured well without bloating the common path.

- Branching rules expand sections (e.g. detailed injury history, equipment nuances) based on prior answers.
- The expansion logic is configurable so the onboarding-depth-vs-speed tradeoff can be tuned.

### E2-S10 — Benchmark-lift calibration `[P1]`
As an experienced **Trainee**, I want to enter and update benchmark lifts over time, so that the coach calibrates loads precisely.

- Benchmark lifts can be added/updated and are versioned with dates.
- Updated benchmarks influence subsequent generation.

### E2-S11 — Bodyweight trend tracking `[P1]`
As a **Trainee**, I want my bodyweight tracked over time, so that the coach and I can see trends.

- Repeated weight entries form a timeline with a simple trend view.
- Trend data is available to programming and diet guidance.

## E3 — Coach Memory (enhancements)

### E3-S4 — Coach's running notes `[P1]`
As the **Coach (system)**, I want to keep my own running notes (autoregulation events, plateaus, observations), so that I reason like a coach who remembers.

- The coach writes structured notes during/after sessions into Coach Memory.
- Notes are included in future planning prompts.

### E3-S5 — Surface memory to the user `[P2]`
As a **Returning user**, I want to see what the coach remembers about me, so that I trust it and can correct it.

- A readable view summarizes key memory (goals, active injuries, recent themes, coach notes).
- I can correct or remove items.

## E4 — Recovery & Readiness (enhancements)

### E4-S5 — Multi-wearable support `[P1]`
As a **Trainee**, I want Oura, Whoop, and Garmin data supported via Health Connect, so that I'm not tied to one device.

- Readiness computes correctly from whichever supported source provides signals.
- Conflicting/overlapping sources are reconciled deterministically.

### E4-S6 — Readiness history and trends `[P1]`
As a **Trainee**, I want to see readiness over time, so that I can spot patterns in my recovery.

- A trend view charts readiness and its main drivers over weeks.

### E4-S7 — Manual readiness override `[P2]`
As a **Trainee**, I want to tell the coach I feel better/worse than the data says, so that my felt state is respected.

- I can adjust or annotate readiness; the override feeds generation and is logged.

## E5 — Coaching Engine & Session Generation (enhancements)

### E5-S6 — Multi-week periodization `[P1]`
As a **Returning user**, I want programming that progresses across weeks (including deloads), so that I improve over time rather than session to session.

- Generation accounts for plan position within a periodized block, including scheduled deloads.
- Reasoning references the macro plan when relevant.

### E5-S7 — Plateau detection and re-planning `[P1]`
As a **Returning user**, I want the coach to notice plateaus and adjust, so that stalls are broken.

- The system detects stalls from workout logs and triggers re-planning.
- The adjustment and its rationale are explained to the user.

### E5-S8 — Generation cost and latency optimization `[P1]`
As the **Admin / Operator**, I want generation cost and latency measured and optimized, so that the one-call-per-session model stays viable at scale.

- Cost and latency per generation are measured against targets in real usage.
- Optimizations (prompt sizing, caching, model selection) are validated against quality.

## E6 — In-Session Experience (enhancements)

### E6-S6 — Mid-session exercise swaps `[P1]`
As a **Trainee**, I want to swap an exercise mid-session (equipment taken, pain), so that I can keep training without breaking the plan.

- I can request a swap; the app offers safe alternatives respecting injuries and equipment (validated by the safety layer).
- The swap and reason are logged to Coach Memory.

### E6-S7 — RPE-based autoregulation UI `[P1]`
As a **Trainee**, I want to log RPE and see the next set adjust visibly, so that effort drives the session.

- I can record RPE per set; the UI shows the resulting adjustment to the next set.

### E6-S8 — Hands-free / voice control `[P2]`
As a **Trainee**, I want hands-free logging and timer control, so that I'm not tapping mid-lift.

- Core actions (log set, start rest, next exercise) are reachable via voice or large hands-free controls.

## E7 — Injury & Condition Management (enhancements)

> Note: Injury identification assist (E7-S5) moved to the MVP — see `v1-mvp.md`.

### E7-S6 — Proactive temporal check-ins `[P1]`
As a **Trainee**, I want the coach to check back on a condition after an appropriate interval, so that my plan updates as I heal, like a human coach would.

- After a configurable interval since last update, the coach proactively asks how the condition is doing.
- My response updates status and programming; declining to answer leaves status unchanged.

### E7-S7 — Return-to-normal as injuries resolve `[P1]`
As a **Trainee**, I want the plan to ease back toward normal as a condition resolves, so that I'm not restricted longer than necessary.

- Marking a condition managed/resolved progressively relaxes related contraindications.
- The transition is gradual and explained in reasoning.

## E8 — Healthy-Aging / Longevity Programming (enhancements)

### E8-S3 — VO2max / high-intensity protocols `[P1]`
As an aging **Trainee**, I want deliberate high-intensity cardiovascular work programmed, so that I build the capacity that most affects healthspan.

- The engine programs structured high-intensity/VO2max work according to my emphases and readiness.
- Protocols respect injuries and the safety layer.

### E8-S4 — Bone density and balance programming `[P1]`
As an aging **Trainee**, I want targeted bone-loading and balance work, so that I protect against age-related decline.

- Sessions include bone-loading and balance work scaled to age, level, and emphases.

### E8-S5 — Track healthspan progress `[P2]`
As a **Trainee**, I want to see progress on healthspan capacities over time, so that I know the long-game work is paying off.

- Simple progress indicators for the healthspan emphases are tracked and shown.

### E9-S3 — Equipment capture by photo `[P2]`
As a **Trainee**, I want to add a location's equipment by photographing the gym or a list, so that setup is faster.

### E9-S4 — Travel context assist `[P2]`

## E10 — Classes (Equinox)

**Goal:** Ingest the user's class schedule and let the coach slot classes in when they serve the plan better than a lift. (Spec priority #4; no public gym API, so data is user-provided.)

### E10-S1 — Enter a weekly class schedule `[P1]`
As an **Equinox member**, I want to enter my weekly class schedule, so that the coach knows what classes are available and when.

- I can manually enter classes with name, type, day, and time.
- The schedule is stored in Coach Memory and is editable.

### E10-S2 — Import classes from a PDF or screenshot `[P1]`
As an **Equinox member**, I want to upload a PDF or screenshot of the class schedule, so that I don't have to type it.

- An uploaded PDF/screenshot is parsed into structured class entries I can review and correct.
- Parsing failures fall back gracefully to manual entry.

### E10-S3 — Coach suggests slotting a class `[P1]`
As an **Equinox member**, I want the coach to recommend a class when it serves my plan better than another lift, so that I use the membership I already pay for.

- When a class (e.g. mobility or conditioning) better serves today's goals/readiness/injuries, the session includes an optional class suggestion with reasoning.
- The suggestion respects schedule, location context, and injuries; declining keeps the lifting session intact.

## E12 — Offline, Sync & Caching (enhancements)

### E12-S3 — Cross-device sync conflict resolution `[P1]`
As a **Trainee** using multiple devices, I want edits and logs to merge cleanly, so that I never lose or duplicate data.

- Concurrent edits from multiple devices resolve deterministically with no data loss.

### E12-S4 — Pre-cache next likely session `[P2]`
As a **Trainee**, I want my next session pre-generated when conditions are stable, so that it's instantly ready offline.

- When inputs are unlikely to change, the next session can be pre-generated and cached, and is regenerated if a trigger fires.

## E13 — Safety & Compliance Layer (enhancements)

### E13-S3 — Configurable safety rule set `[P1]`
As the **Admin / Operator**, I want safety contraindications and bounds maintained as configurable rules, so that we can update limits without code changes.

- Rules (contraindications, load/volume bounds) are editable configuration with versioning.
- Changes are auditable.

### E13-S4 — Safety audit trail `[P1]`
As the **Admin / Operator**, I want a queryable audit trail of safety decisions, so that we can investigate incidents and satisfy review.

- Every safety pass/correct/regenerate event is recorded with inputs and outcome, access-controlled.

## E15 — Backend Platform & Observability (enhancements)

### E15-S4 — Quality and cost dashboards `[P1]`
As the **Admin / Operator**, I want dashboards for generation quality, latency, and cost, so that I can manage the LLM economics in production.

- Dashboards track per-session cost, latency, and quality signals over time.

### E15-S5 — Onboarding-depth experimentation `[P2]`
As the **Admin / Operator**, I want to experiment with onboarding depth and cold-start quality, so that we resolve the depth-vs-speed tradeoff with data.

As the **Admin / Operator**, I want monitoring of ingestion health across sources, so that signal gaps are caught early.

- Ingestion success/failure and data freshness are monitored per source with alerting on anomalies.
