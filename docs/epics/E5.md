# E5 — Coaching Engine & Session Generation

**Goal:** Generate the full session at "Start workout," explain it, cache it for offline use, and
re-plan on triggers. **Wave 2 — the integration hub.** Depends on: E3, E13, E4, E7, E9, E2.
Parallel with: E8 (built together). Keep mostly single-stream for coherence.

> Provide a **fixed sample session JSON** early so the Android session-preview UI (and E6) can build in parallel.

Stories: E5-S1 generate on demand, E5-S2 reasoning, E5-S3 cache offline, E5-S4 on-device
autoregulation, E5-S5 `[P1]` re-plan on triggers.

## PRs

### E5-PR1 — Session schema & sample JSON `[contract]` — *E5-S1, E5-S3*
- Define the session output schema in `backend/api/`: warmup, main work (loads/reps + explicit
  **rest periods**), accessory, **aging/mobility block** (E8), and reasoning. Commit a fixed sample JSON.
- Tests: schema validates the sample.
- Done: client/UI can build against a stable shape.

### E5-PR2 — Claude client `[backend]` — *E5-S1*
- `internal/coaching` Anthropic client using the server-side key (E15 config), with timeout/retry;
  **client-side never calls Anthropic.** Model id from config.
- Tests: client builds requests correctly (mock transport); key sourced from config only.
- Done: a server-side Claude call path exists.

### E5-PR3 — Generate full session `[backend]` — *E5-S1, E5-S2*
- Assemble via E3 (memory + readiness + injuries + location/equipment + plan position) → call Claude →
  parse into the session schema, including **brief plain-language reasoning** (no medical claims).
- Tests: assembly→call→parse produces a schema-valid session; reasoning present and claim-free.
- Done: one call at "Start workout" yields a full session grounded in current state.

### E5-PR4 — Safety validation in the loop `[backend]` — *E5-S1*
- Run every generated plan through `safety.Validate` (E13-PR4 / E7-PR5); correct or **regenerate**
  on violation; never return an invalid plan. Log generation + safety outcomes (E15-S3).
- Tests: a violating generation is corrected/regenerated before return; outcome logged.
- Done: only validated sessions are delivered.

### E5-PR5 — Cache the session for offline `[android]` — *E5-S3*  ⚠️ NOT DONE (Wave 3)
- Persist the full returned session (sets, rests, timers, reasoning) in `core/db` so it runs with no
  signal; in-session logging works offline and queues for sync (handshake with E12).
- Tests: session readable offline; logs queue when offline.
- Done: the whole session is available offline once generated.
- **Status:** deferred to Wave 3. The E5-S3 client handshake shipped on `wave2-e5-e8` — a
  read-only `feature/session` preview, the `SessionDto` family, and a strict contract test against
  the committed sample — so this is just the Room persistence + sync-queue layer on top.

### E5-PR6 — On-device autoregulation `[android]` — *E5-S4*  ⚠️ NOT DONE (Wave 3)
- Local logic in `feature/session` that adjusts the next set toward target RPE from logged
  performance — **no LLM call, works offline.**
- Tests (TDD): given a logged set vs. target RPE, the next set adjusts as specified; runs offline.
- Done: sets adapt locally without the server.
- **Status:** deferred to Wave 3. Unblocked: each set carries a `rpe_target` in the published schema
  for the local adjustment logic to read.

### E5-PR7 — Re-plan triggers `[backend]` `[P1]` — *E5-S5*
- Re-plan on next session by default; **immediately** on a trigger: new/changed injury (E7-PR3),
  poor-recovery morning (E4-PR4), or context change (E9-PR2). Plateau-driven re-plan deferred to V1.1.
- Tests: each trigger forces a regenerate; absent triggers defer to next session.
- Done: the plan stays current on material change.

## Definition of done (epic)
Tapping "Start workout" returns a validated, cached, offline-ready session with reasoning and an
aging block; autoregulation runs on-device; re-planning triggers fire.
