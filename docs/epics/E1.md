# E1 — Authentication, Accounts & Privacy

**Goal:** A secure account that anchors cloud-backed Coach Memory across devices, with clear
consent for sensitive health data. **Wave 0.** Depends on: E15 (skeleton). Parallel with: E3 schema design.

Stories: E1-S1 signup, E1-S2 login/logout, E1-S3 password reset, E1-S4 health-data consent,
E1-S5 account+data deletion, E1-S6 multi-device session restore. (All `[P0]`.)

## PRs

### E1-PR1 — Accounts schema & user model `[backend]` `[contract]` — *E1-S1*
- Migration for `users` (id, email unique, password_hash, email_verified, timestamps). Define the
  auth API contract in `backend/api/` (signup/login/refresh/logout/reset/consent/delete).
- Tests: migration applies; email uniqueness enforced.
- Done: schema + contract reviewed before handler work.

### E1-PR2 — Signup with validation `[backend]` — *E1-S1*
- `POST /auth/signup`: email-format + password-strength validation; bcrypt/argon2 hash;
  **non-enumerating** duplicate response; issue session on success.
- Tests: valid signup creates+signs in; weak password rejected; duplicate email gives generic error.
- Done: account created and authenticated; no enumeration.

### E1-PR3 — Login, logout & token sessions `[backend]` — *E1-S2, E1-S6*
- `POST /auth/login` (generic failure on bad creds), `POST /auth/logout` (revoke refresh),
  `POST /auth/refresh`. Access + refresh tokens (refresh persisted/revocable for multi-device).
- Tests: correct creds authenticate; wrong creds generic; logout revokes; refresh rotates.
- Done: sessions issued/revoked; multi-device restore possible via login on a new device.

### E1-PR4 — Rate limiting / backoff on auth `[backend]` — *E1-S2*
- Per-account + per-IP backoff on repeated failed logins/reset requests.
- Tests: Nth failed attempt is throttled; success resets the counter.
- Done: brute-force attempts are slowed.

### E1-PR5 — Password reset `[backend]` — *E1-S3*
- `POST /auth/reset/request` (always 200, no enumeration; time-limited single-use token) and
  `POST /auth/reset/confirm`. Successful reset **invalidates other sessions**.
- Tests: token single-use + expiry; reset revokes existing refresh tokens; unknown email still 200.
- Done: recoverable access; reset kills other devices' sessions.
- *Email delivery may be stubbed/logged in MVP; note the seam.*

### E1-PR6 — Health-data consent capture `[backend]` `[contract]` — *E1-S4*
- Migration `consents` (user_id, type, version, accepted_at). `GET/POST /consent`. Stores the
  disclaimer version accepted. Coordinates with E13 disclaimer text.
- Tests: consent persists with version; absence is queryable (gates Health Connect reads in E4).
- Done: consent state recorded and retrievable.

### E1-PR7 — Account & data deletion `[backend]` — *E1-S5*
- `DELETE /account`: confirm, then cascade-delete profile, Coach Memory, history, ingested health
  data; emit completion. Honors E15-S2 storage and E3 data.
- Tests: deletion removes all user-owned rows across tables; idempotent; returns confirmation.
- Done: backend data actually gone after deletion.

### E1-PR8 — Android auth scaffold & secure token storage `[android]` — *E1-S1, E1-S2, E1-S6*
- `feature/auth` screens (signup/login/logout) + `core/auth` token storage backed by Android
  Keystore. Session persists across app restarts; logout clears local tokens + cached personal data.
- Tests: ViewModel state transitions; token persistence; logout wipes cache.
- Done: user can sign up/log in/out against the backend; session survives restart.

### E1-PR9 — Android consent & deletion UI `[android]` — *E1-S4, E1-S5*
- Pre-Health-Connect consent screen with plain-language disclosure + medical disclaimer
  (E13 text); manual-only mode if declined. Settings entry to delete account with confirmation.
- Tests: declining consent routes to manual mode; deletion requires confirmation.
- Done: consent gating + deletion flow wired end-to-end.

## Definition of done (epic)
Sign up / log in / log out / reset work; sessions are secure & multi-device; consent captured;
account deletion removes backend + local data. Disclaimer language present at consent.
