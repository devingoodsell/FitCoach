# E3 — Coach Memory

**Goal:** A durable, structured, **versioned** store the model reads on every decision, plus a
deterministic prompt-assembly function. **Wave 0.** Depends on: E15, E1 (identity). Gates Wave 1.

Stories: E3-S1 persist structured memory, E3-S2 assemble memory into every planning call,
E3-S3 record session outcomes back to memory. (All `[P0]`.)

## PRs

### E3-PR1 — Coach Memory schema & versioning `[backend]` `[contract]` — *E3-S1*
- Migrations for the memory tables (profile, goals, schedule, preferences, locations, injuries,
  diet, workout_logs, coach_notes — owned by `user_id`) plus a **schema_version** stored with each
  record so it can evolve without data loss. Define the memory read/write contract in `backend/api/`.
- Tests: migration applies; version column populated; FK cascade aligns with E1-PR7 deletion.
- Done: schema + version strategy reviewed before code.

### E3-PR2 — Memory store: read/write per domain `[backend]` — *E3-S1*
- `internal/memory` repository with typed get/upsert per domain section. Cloud-backed, account-scoped.
- Tests: round-trip each section; version preserved; cross-user isolation enforced.
- Done: backend can read/write a versioned Coach Memory record tied to an account.

### E3-PR3 — Schema migration/upgrade path `[backend]` — *E3-S1*
- An in-code upgrader that reads an older `schema_version` record and migrates it forward on read.
- Tests: a vN-1 record loads and upgrades to vN with no data loss.
- Done: evolving the schema doesn't strand old data.

### E3-PR4 — Deterministic prompt assembly `[backend]` — *E3-S2*
- `AssemblePrompt(ctx, userID, currentState)` gathering profile, goals, active/managed injuries,
  recent history, readiness, location/equipment, relevant preferences into a stable, ordered prompt
  payload. **Deterministic** (same inputs → same output) and logged (redacted) for debugging.
  Model call itself is stubbed here (lands in E5).
- Tests: deterministic output for fixed inputs; secrets/PII redacted in the debug log; missing
  sections degrade gracefully.
- Done: a single function produces the planning payload other epics build on.

### E3-PR5 — Record session outcomes `[backend]` — *E3-S3*
- Write completed sessions (exercises, reps, weight, rests, timed work) and autoregulation events
  into `workout_logs`. Consumed later by E5/E6/E12.
- Tests: a completed session persists; autoregulation events captured; idempotent write (ties E12).
- Done: future sessions can build on what was actually done.

### E3-PR6 — Android memory cache & repository `[android]` — *E3-S1, E1-S6*
- `data/` repositories + `core/db` (Room) caching memory locally for offline reads; sync on login
  (multi-device restore). No user data hard-coded — all loaded from the account.
- Tests: cache round-trip; fresh-device login hydrates from backend.
- Done: memory follows the user across devices; available offline.

## Definition of done (epic)
Versioned Coach Memory persists all domain sections, upgrades old records safely, assembles
deterministically into a planning payload, records outcomes, and is cached/portable on the client.
