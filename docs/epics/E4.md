# E4 — Recovery & Readiness (Health Connect)

**Goal:** Ingest recovery signals through Health Connect and compute our **own** readiness, then
feed it into programming. **Wave 1.** Depends on: E3. Parallel with: E2, E7, E9, E11, E13.
Split: heavy Android Health Connect piece + backend readiness-compute piece.

Stories: E4-S1 connect Health Connect, E4-S2 compute in-house readiness, E4-S3 show readiness +
explanation, E4-S4 readiness drives the session. (All `[P0]`.)

## PRs

### E4-PR1 — Signals contract & storage `[backend]` `[contract]` — *E4-S1, E4-S2*
- `backend/api/` for uploading raw signals (sleep, RHR, HRV samples) + reading readiness. Migration
  for a raw-signals table (account-scoped, timestamped). Gated by E1 consent.
- Tests: signals persist; upload rejected without consent.
- Done: contract + storage fixed for both tracks.

### E4-PR2 — In-house readiness computation `[backend]` — *E4-S2*
- `internal/readiness`: compute score from raw signals (overnight **HRV trend**, **resting HR
  trend**, **sleep duration & quality**) vs. a rolling baseline. **Not** a vendor score. Handles
  missing/partial data and **flags low-confidence days**. TDD with fixtures.
- Tests: known fixtures → expected scores; missing-signal cases don't crash and flag low confidence;
  trend math correct.
- Done: deterministic readiness from raw signals.

### E4-PR3 — Readiness explanation `[backend]` — *E4-S3*
- Produce a short plain-language driver explanation (e.g. "HRV down vs. baseline, short sleep")
  with **no medical claims**. Centralized phrasing.
- Tests: explanation names the dominant drivers; passes a no-medical-claims check.
- Done: explanation accompanies each score.

### E4-PR4 — Readiness feeds generation `[backend]` — *E4-S4*
- Expose readiness through E3 prompt assembly so it's an input to volume/intensity/selection; a
  poor-recovery morning is a **re-planning trigger** (handshake with E5-S5).
- Tests: assembly includes today's readiness; poor-recovery flag surfaces as a trigger.
- Done: readiness is wired as a generation input (E5 consumes it).

### E4-PR5 — Health Connect integration `[android]` — *E4-S1*  ⚠️ NOT DONE (deferred)
- `healthconnect/`: request permissions (sleep, RHR, HRV) **with rationale**; read at least Pixel
  Watch data; **graceful degrade** to manual/no-readiness mode on denial or no data. Gated by E1 consent.
- Tests: permission-denied path degrades; ingestion maps to the upload contract.
- Done: app reads recovery signals and uploads them.
- **Status:** deferred. The backend (signals storage `POST /health/signals`, readiness compute,
  `GET /readiness`) and the readiness UI (E4-PR6) with its no-data fallback are done and on `main`.
  This PR is the only remaining E4 piece: it needs the `androidx.health.connect:connect-client` SDK
  and **on-device** verification (permission flow + Pixel Watch reads), which can't be done in the
  headless build. Wire the read→`Sample`→`POST /health/signals` upload (keep the record→value
  aggregation in pure, unit-testable functions) and the permission-request UX.

### E4-PR6 — Readiness UI `[android]` — *E4-S3*
- `feature/readiness`: show today's score + explanation; manual/no-readiness fallback state.
- Tests: score+explanation render; fallback state shown when unavailable.
- Done: user sees readiness with a plain-language why.

## Definition of done (epic)
App connects Health Connect (or degrades), backend computes its own readiness handling gaps,
explains it without medical claims, and readiness is an input to session generation.
